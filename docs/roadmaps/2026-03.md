# Ronin — March 2026 Roadmap

## North Star
Ronin understands agent *systems*, not just individual agents. George can export his CEO Operating System constellation from his personal account and import it into his work account — with path rewriting, secret stripping, memory bootstrapping, and provenance tracking.

## Milestones

### M1: Constellation Data Model — Complete
- **Why it matters**: Ronin currently sees atoms (individual passports). It needs to see molecules — systems of agents, skills, hooks, and config that work together as a unit.
- **Acceptance criteria**:
  - [x] Define constellation schema: name, description, author, components (passports + non-passport files like hooks, memory schemas, CLAUDE.md patterns), inter-component dependencies
  - [ ] `ronin constellation detect` — heuristic detection from existing inventory (deferred to polish phase)
  - [x] `ronin constellation define "ceo-operating-system"` — manual constellation creation, selecting components
  - [x] `ronin constellation list` and `ronin constellation show <name>` — view constellations and their component graph
  - [x] SQLite schema: `constellations` table, `constellation_components` join table
  - [ ] Dashboard: constellation grouping in the Yard (deferred to polish phase)
- **Key files**: `src/db.ts`, `dashboard/lib/db.ts`, `src/scanner.ts`

### M2: Export/Import Engine — Not Started
- **Why it matters**: This is the actual porting experience. George needs to move his CEO system to his work machine this month.
- **Acceptance criteria**:
  - [ ] `ronin export <constellation>` produces a portable bundle directory with manifest.json
  - [ ] Path rewriting: absolute paths → parameterized `{{PROJECT_ROOT}}`, `{{HOME}}`, etc.
  - [ ] Secret stripping: memory.json people entries, .env references, project-specific data flagged for review or excluded
  - [ ] Memory schema bootstrapping: export the *structure* of memory.json (empty projects, empty people) not the *data*
  - [ ] `ronin import <bundle>` installs into target `~/.claude/` with guided setup (project root, org name, etc.)
  - [ ] Dependency check: flag required MCP servers, tools, or npm packages
  - [ ] Dry-run mode: show what would be created/modified before writing
  - [ ] Git-based transfer: export produces a git-friendly directory (no binaries, clean .gitignore)
- **Key files**: new `src/export.ts`, `src/import.ts`

### M3: Provenance & Licensing Metadata — Not Started
- **Why it matters**: When agents move between accounts/machines, provenance answers "where did this come from and am I allowed to use it here?" This is the seed of the open Passport spec.
- **Acceptance criteria**:
  - [ ] Passport schema extended: `origin` (account/machine ID), `author`, `license` (personal, work, open, commercial), `forkedFrom`
  - [ ] `ronin export` embeds provenance in manifest: who exported, when, from where, license terms
  - [ ] `ronin import` preserves provenance: dashboard shows origin and license
  - [ ] `ronin constellation sync` — detect when source has been updated, show diff (read-only, no auto-merge)
- **Key files**: Proto-Passport schema, `src/export.ts`, `dashboard/`

### M4: Dog-food — Port the CEO System — Not Started
- **Why it matters**: If George can't port his own constellation smoothly, nobody can. Real usage on two machines reveals every gap.
- **Acceptance criteria**:
  - [ ] Scan work machine's `~/.claude/` via authed git repo — map the target environment
  - [ ] Define "ceo-operating-system" constellation from current csuite components (cofounder, chief-of-staff, standup, roadmap, debrief, leadership-meeting, panel hooks, memory schema, CLAUDE.md patterns)
  - [ ] Export from personal account
  - [ ] Transfer via git to work machine
  - [ ] Import on work machine with path rewriting and memory bootstrapping
  - [ ] Verify: standup, roadmap, cofounder, debrief all functional in work context
  - [ ] Reverse direction: identify one work constellation candidate to port home
  - [ ] Document all friction → feed back into M1-M3
- **Linear tickets**: none yet (ronin has no Linear team)

### M5: Constellation-Aware Roster — Not Started
- **Why it matters**: The roster hook (already running in every session) should surface constellation context, not just individual agents.
- **Acceptance criteria**:
  - [ ] `ronin roster --hook` output groups agents by constellation when applicable
  - [ ] `/roster` skill shows constellation membership in results
  - [ ] `ronin stats` includes constellation counts

## Deferred (explicitly not March)
- **npm publish / community launch** — ship with constellations, not without. The flat inventory tool undersells the vision.
- **Translation validation (image generation)** — can happen anytime, not blocking core work
- **Test suite / CI** — important but not before constellation features exist to test
- **Linear team creation** — do this when launching publicly, not before

## Dependencies
- George's work machine git repo access (needed for M4 target environment scan)
- No external service dependencies — everything is local-first

## Risks
- **Export/import path rewriting complexity** — absolute paths are everywhere (hooks, skills, CLAUDE.md). Mitigation: start with the CEO system as the only test case, parameterize incrementally.
- **Constellation detection heuristics** — auto-detect may be unreliable. Mitigation: manual `define` is the primary flow, `detect` is nice-to-have.
- **Work environment differences** — different OS, directory layout, or Claude Code version. Mitigation: M4 is explicitly the dog-food milestone to surface these.
- **Scope creep on provenance** — licensing metadata could become arbitrarily complex. Mitigation: keep it to 4 license types and a simple origin field for now.
