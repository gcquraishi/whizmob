# Whizmob — March 2026 Roadmap

## North Star
Whizmob understands agent *systems*, not just individual agents. George can export his CEO Operating System mob from his personal account and import it into his work account — with path rewriting, secret stripping, memory bootstrapping, and provenance tracking.

## Milestones

### M1: Mob Data Model — Complete
- **Why it matters**: Whizmob currently sees atoms (individual passports). It needs to see molecules — systems of agents, skills, hooks, and config that work together as a unit.
- **Acceptance criteria**:
  - [x] Define mob schema: name, description, author, components (passports + non-passport files like hooks, memory schemas, CLAUDE.md patterns), inter-component dependencies
  - [ ] `whizmob mob detect` — heuristic detection from existing inventory (deferred to polish phase)
  - [x] `whizmob mob define "ceo-operating-system"` — manual mob creation, selecting components
  - [x] `whizmob mob list` and `whizmob mob show <name>` — view mobs and their component graph
  - [x] SQLite schema: `constellations` table, `constellation_components` join table
  - [ ] Dashboard: mob grouping in the inventory (deferred to polish phase)
- **Key files**: `src/db.ts`, `dashboard/lib/db.ts`, `src/scanner.ts`

### M2: Export/Import Engine — Complete
- **Why it matters**: This is the actual porting experience. George needs to move his CEO system to his work machine this month.
- **Acceptance criteria**:
  - [x] `whizmob export <mob>` produces a portable bundle directory with manifest.json
  - [x] Path rewriting: absolute paths → parameterized `{{HOME}}`, `{{CLAUDE_DIR}}`, `{{WHIZMOB_DIR}}`
  - [x] Secret stripping: tokens, keys, passwords redacted; .mcp.json env blocks sanitized
  - [x] Memory schema bootstrapping: export the *structure* of memory.json (empty projects, empty people) not the *data*
  - [x] `whizmob import <bundle>` installs into target `~/.claude/` with parameter overrides (`--param`)
  - [x] Dependency check: flags required MCP servers and npm packages
  - [x] Dry-run mode: show what would be created/modified before writing
  - [x] Git-based transfer: export produces a git-friendly directory (no binaries, clean .gitignore)
- **Key files**: new `src/export.ts`, `src/import.ts`

### M3: Provenance & Licensing Metadata — Complete
- **Why it matters**: When agents move between accounts/machines, provenance answers "where did this come from and am I allowed to use it here?" This is the seed of the open Passport spec.
- **Acceptance criteria**:
  - [x] Passport schema extended: `origin` (account/machine ID), `author`, `license` (personal, work, open, commercial), `forkedFrom`
  - [x] `whizmob export` embeds provenance in manifest: who exported, when, from where, license terms
  - [x] `whizmob import` preserves provenance: writes provenance back to local DB
  - [x] `whizmob mob sync` — detect when source has been updated, show diff (read-only, no auto-merge)
- **Key files**: Proto-Passport schema, `src/export.ts`, `dashboard/`

### M4: Dog-food — Port the CEO System — In Progress
- **Why it matters**: If George can't port his own mob smoothly, nobody can. Real usage on two machines reveals every gap.
- **Acceptance criteria**:
  - [ ] Scan work machine's `~/.claude/` via authed git repo — map the target environment
  - [x] Define "ceo-operating-system" mob from current csuite components (12 components: cofounder, chief-of-staff, standup, roadmap, debrief, leadership-meeting, sprint, sprint-coordinator, panel hooks, memory schema, CLAUDE.md patterns)
  - [x] Export from personal account (at `~/.whizmob/exports/ceo-operating-system`)
  - [ ] Transfer via git to work machine
  - [ ] Import on work machine with path rewriting and memory bootstrapping
  - [ ] Verify: standup, roadmap, cofounder, debrief all functional in work context
  - [ ] Reverse direction: identify one work mob candidate to port home
  - [ ] Document all friction → feed back into M1-M3
- **Blocked**: Remaining criteria require George's work machine access
- **Tickets**: none yet (whizmob has no dedicated ticket prefix)

### M5: Mob-Aware Roster — Complete
- **Why it matters**: The roster hook (already running in every session) should surface mob context, not just individual agents.
- **Acceptance criteria**:
  - [x] `whizmob roster --hook` output groups agents by mob when applicable
  - [x] `/roster` skill shows mob membership in results
  - [x] `whizmob stats` includes mob counts

## Deferred (explicitly not March)
- **npm publish / community launch** — ship with constellations, not without. The flat inventory tool undersells the vision.
- **Translation validation (image generation)** — can happen anytime, not blocking core work
- **Test suite / CI** — important but not before constellation features exist to test
- **Linear team creation** — do this when launching publicly, not before

## Dependencies
- George's work machine git repo access (needed for M4 target environment scan)
- No external service dependencies — everything is local-first

## Risks
- **Export/import path rewriting complexity** — absolute paths are everywhere (hooks, skills, CLAUDE.md). Mitigation: start with the CEO system as the only test case, parameterize incrementally.
- **Constellation detection heuristics** — auto-detect may be unreliable. Mitigation: manual `define` is the primary flow, `detect` is nice-to-have.
- **Work environment differences** — different OS, directory layout, or Claude Code version. Mitigation: M4 is explicitly the dog-food milestone to surface these.
- **Scope creep on provenance** — licensing metadata could become arbitrarily complex. Mitigation: keep it to 4 license types and a simple origin field for now.
